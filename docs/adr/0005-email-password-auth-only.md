# 인증은 이메일+비밀번호만 (소셜·비밀번호 재설정·MFA 후속)

SPA에 인증 계층을 처음 도입하는 이번 wave는 Supabase Auth의 **이메일+비밀번호**만 쓴다. 확인메일은 끈다(`auth.email.enable_confirmations = false`) — 회원가입 즉시 세션이 발급되고 바로 로그인 상태가 된다. 소셜/OAuth 로그인, 비밀번호 재설정·찾기, MFA, 이메일 변경은 **후속 wave**로 미룬다. anon 키를 쓰고 RLS가 인가 경계라는 전제(ADR-0001)는 그대로다.

## 검토한 대안

- **OAuth(소셜) 지금 도입** — 기각: 프로바이더 설정·콜백 라우트·계정 연결이라는 표면이 더 붙는데 지금 필요 없다. config에 프로바이더도 안 붙어 있다.
- **비밀번호 재설정 지금 포함** — 기각: 이메일 발송 흐름이 필요하고, 이번 9개 화면에 없다.

## 결과

- 비밀번호를 잊은 사용자는 현재 복구 경로가 없다(후속까지). 초기/로컬 단계에서는 수용한다.
- 회원가입 직후 JWT에 이미 `member_id`·roles·status가 실린다(트리거 커밋 이후 토큰 발급 — 실증 확인). 따라서 가입 직후 글쓰기에 별도 토큰 갱신이 필요 없다(ADR-0003의 지연은 가입 플로우엔 무관).
- 소셜·비번재설정을 도입할 때 이 ADR을 갱신한다.
